Diário de bordo

Pontos de evolução do MMB

Registro humano-legível de marcos do método. Diferente do changelog (lista de commits), do journal.jsonl (eventos máquina-legíveis) e dos .tooling/designs/* (decisões técnicas por versão). Aqui mora o saldo reflexivo: o que mudou, o que aprendemos, o que ficou aberto.

Sobre este doc canônico

Diário de evolução do método (não dos produtos sobre o método). Cada entry corresponde a um marco significativo — tag git, virada de fase, fim de uma esteira corretiva, ou descoberta de padrão que muda como o andaime opera.

Não substitui:

  • .tooling/logs/journal.jsonl — eventos estruturados, consumidos por review-cycle.sh ao fechar épico.
  • .tooling/designs/v0.1-comm-robustness.md, v0.2-journal-and-review.md — design docs por versão (técnico).
  • .tooling/intents/<date>-<slug>/master-briefing.md — nota narrativa final por épico (local, granular).
  • Tag annotations no git — release notes curtas, bullets.

Convenção de updates

  • Quando atualizar: ao fim de marco significativo — não por PR, não por épico individual.
  • Quem atualiza: Mestre interativo, ao fim do marco, mostrando pro Rick antes de commitar.
  • Ordem: cronológica reversa (mais recente no topo).
  • Forma: saldo (números) + o que mudou (3-5 bullets concretos) + o que aprendemos (1-3 insights reflexivos) + débitos abertos com gatilhos explícitos.
  • Não: diário de conversa, atualização contínua por commit, registro de cada bug. Para isso, há outras peças.

v0.11.0 — Captura de modelo Claude por ciclo

v0.11.0 2026-05-18 · pós-tag v0.10.0 · sessão 2026-05-18

Épico curto e linear (logger-model-tracking): fechar o circuito de observabilidade do modelo Claude usado em cada ciclo. Antes desta janela, cost_usd e contagens de tokens eram persistidos no mmb-logger sem o eixo principal de variação — opus vs sonnet vs haiku é ordem de grandeza em $/token, e a info estava sendo descartada após o spawn de cada agente. Três tarefas serializadas (andaime → schema/ingestion → backfill) cobriram o gap e desbloqueiam o experimento futuro de modelo menor em atômicos.

Saldo

1 commit no mmb-andaime + 2 PRs no mmb-logger (#20 schema/ingestion/API, #22 backfill)
1 épico entregue (logger-model-tracking, fechado last_in_epic:true)
19 ciclos retroagidos com claude-opus-4-7 via backfill-model + 1 warning estruturado no journal (ciclo bootstrap pré-janela MMB_MODE)
~1h21min do dispatch ao task-fechada-L4 · 0 retries, 0 questions/errors do orq do logger

O que mudou de fato

T1 — andaime (commit fad54fb):

  • agents.sh register aceita 6º argumento opcional [model]; _build_json já tolerava kv extra, então compat retroativa é total.
  • up.sh, spawn-atomic.sh e worker.sh passam MMB_MODEL_* resolvido por camada nos eventos spawn de state/agents.jsonl.
  • Workers stateless ganharam register/deregister em par via trap EXIT; id <dest>-w-<pid> evita colisão com o master interativo e entre invocações.

T2 — mmb-logger (PR #20):

  • ALTER TABLE ciclos ADD COLUMN model TEXT, migration idempotente via init-db.
  • Ingestion lê model dos spawn events do planner do projeto e popula ciclos.model; tolerante a ausência (NULL silencioso) pra eventos legacy.
  • API expõe model no payload dos ciclos. 17 testes novos em test_reconcile_model.py.

T3 — mmb-logger (PR #22):

  • Sub-pacote backfill/ + CLI mmb-logger backfill-model [--dry-run]. One-shot idempotente que lê git log -- .tooling/config.sh no andaime, monta janelas temporais por default de MMB_MODE, e mapeia ciclos pré-T1 pro modelo do planner default daquela janela.
  • Ciclos sem janela válida ficam NULL com warning estruturado backfill-model-ambiguous no journal (deduplicado por cycle_id em re-runs).
  • Domínio respeitado: UPDATE só toca model; assertiveness_score/review_note (campos humanos) intocados por construção. 14 testes novos em test_backfill_model.py.

O que aprendemos

  • Mestre edita andaime sob aviso explícito funciona. T1 não tinha orq (andaime não é repo de produto), então o Mestre executou diretamente — Constraint #2 do master.md permite com "aviso explícito", e o briefing mestre fez esse papel. Sem cerimônia extra de orq fantasma.
  • Post-merge é manual no modelo stateless. Orq stateless não detecta merge externo: a sessão acordou via commd só quando o Mestre disparou briefing curto de continuação ("PR mergeado, faça task-end + materialize próxima"). Esse padrão repetiu duas vezes no épico (após PR #20 e após PR #22). Não virou frente de automação ainda — manual é barato e raro o suficiente.
  • Dry-run real bateu com run real. 19 candidatos backfilled como claude-opus-4-7, 1 ambíguo, em ambas as execuções. Idempotência forte (filtro WHERE model IS NULL no UPDATE + dedup via leitura do journal) deu segurança operacional pra rodar em produção sem cerimônia.

Validação em produção

  • Captura ao vivo: spawn events de agents.jsonl pós-T1 incluem o campo model em todos os agentes (master, atômicos, workers stateless). Verificado em logger-L3 e logger-L4, ambos com claude-opus-4-7.
  • Backfill em produção: 19/20 candidatos retroagidos. O 1 ciclo bootstrap cleanup-task-scripts__core__2026-05-14T20:19:30Z ficou NULL com warning estruturado (reason: no-window-match) — anterior ao primeiro commit que introduziu MMB_MODE em config.sh.
  • API: endpoint /api/epicos/<id> voltou a 200 após init-db aplicar a migration no DB local (o --reload do uvicorn retomou sozinho).

Bugs novos

Nenhum bug novo. O review-cycle.sh capturou 1 evento não-resolvido (commd-worker-exit às 13:04:58Z, watchdog matou worker logger por stale_seconds=121), mas o trabalho útil já tinha sido entregue por worker concorrente (issue #19 criada às 13:06). Padrão já documentado em worker.sh ("claude -p tem latência de finalização pós-output que pode passar de 60s"). Resolvido por nota via log.sh warn watchdog-kill-borderline --resolves 20260518T130458Z-logger-worker-watchdog-kill-19779 (evento e-2026-05-18-001). Sem nova frente de hardening — débito 2c em worker.sh (kill -0 $CLAUDE_PID no lugar de log mtime) continua aberto, sem evidência de impacto novo.

Débitos abertos

Débito Gatilho pra atacar Status
Experimento "modelo menor em atômicos" (MMB_MODE=balanced ou override pontual MMB_MODEL_ATOMIC=claude-sonnet-4-6) Infra de captura por camada agora pronta — falta só a decisão operacional. Comparar cost_usd × model agregados pelo logger após N épicos em modo experimental pendente
Watchdog hardening 2c: detectar claude vivo via kill -0 $CLAUDE_PID em vez de log mtime Já documentado em worker.sh; reabrir se eventos worker-watchdog-kill virarem recorrentes pós-trabalho-útil adiado

v0.10.0 — B1 fechado em produção + ciclo iterativo validado

v0.10.0 2026-05-18 · pós-tag v0.9.0 · sessão contínua 2026-05-17 → 2026-05-18

Janela curta (~12h) que validou em produção real as fortificações da v0.9.0, descobriu a causa raiz do bug recorrente B1 (que sobreviveu a 3 tentativas anteriores de fix), entregou 3 épicos de produto pelo método andaime, e ergueu um diário humano-legível (este próprio doc) como peça canônica do método.

Saldo

6 PRs no mmb-andaime + 1 PR no mmb-logger
3 épicos de produto entregues (ux-toggle-e-role-meeseeks, epico-idle-badge; primeiro completou também o cockpit-ux-layout-filtros pendente da janela anterior)
Suíte tooling: 252 → 292 verdes (+40 testes novos)
Suíte mmb-logger: 167 → 172 verdes
B1 fechado oficialmente em produção real após 3 implementações erradas e 1 correta
DIARIO.html introduzido como peça canônica

O que mudou de fato

Diário institucional (este doc):

  • DIARIO.html criado na raiz (PR andaime#10). Sidebar TOC sticky, dark/light, navegável. Diferente de journal.jsonl (eventos máquina) e dos designs/* (técnico por versão). Aqui mora o saldo reflexivo por marco — produto da observação de que .md técnico não atende leitura humana fluente.

Esteira corretiva pós-tag:

  • Wrapper obrigatório send-status-pr-opened.sh (PR andaime#11 — B1.1). Auto-detect de suite_status via gh pr view; ausente honestamente se gh falha.
  • Cleanup automático de heartbeats órfãos em task-end/task-abort (PR andaime#12). Helper mmb_remove_orphan_heartbeat em config.sh.
  • up.sh sobe logger-api com --reload pra dev local (PR andaime#13). CLI já tinha a flag; faltava só usar.
  • Documentação do --reload em mmb-logger/CLAUDE.md (PR logger#18).
  • B1.2: open-pr.sh agora chama send-status-pr-opened.sh ao invés de msg.sh direto (PR andaime#14). Esta foi a causa raiz real do B1.

Épicos de produto entregues nesta janela:

  • cockpit-ux-layout-filtros — scroll local + sidebar retrátil + filtro de tags incluindo épicos (cockpit#20 + logger#17).
  • ux-toggle-e-role-meeseeks — toggle reposicionado no Header + planners amarelos no aquário via módulo agent-role.ts agnóstico (cockpit#22 + aquarium#17).
  • epico-idle-badge — badge distingue 3 estados (aberto/idle/fechado) derivado client-side (cockpit#24). Foi a frente de validação real de B1.2.

O que aprendemos

Quando bug recorrente parece ignorar o profile, primeiro suspect não é "claude do worker está ignorando texto" — é caminho de código alternativo invisível. B1 foi tratado 3 vezes errado:
  • PR andaime#7 reforçou o profile com warning ⚠ + tabela + receita executável + caso real.
  • PR andaime#11 criou wrapper obrigatório.
  • Reincidiu nos 2 workers do próximo épico.
A diagnose só veio na 3ª reincidência: o status pr-aberto-N é emitido por open-pr.sh do atômico (bash hardcoded antes do schema v0.4+), não pelo orq local. Toda a esteira anterior tratou o sintoma errado. Lição transferível: bug recorrente "ignora profile" = procurar o caminho que não passa pelo profile.
Critério acionável pré-registrado fecha o ciclo sem debate. O PR andaime#7 (reforço do profile) já deixava escrito: "se B1 falhar de novo, promover pra wrapper obrigatório". Quando reincidiu, não houve discussão sobre "será que vale wrapper?" — o gatilho estava documentado. Padrão a replicar: anotar critério de escalação no próprio PR que o introduz.
Validação em produção real é diferente de suíte verde. B1.1 (PR #11) passou 22/22 testes mas não tinha chamador real até B1.2 (PR #14) conectar o open-pr.sh. Testes de unidade do wrapper nunca detectariam isso — o gap morava em quem chama, não no chamado. Reforça a importância de smoke real (próximo merge pelo método) como critério complementar a "suite verde".
O método foi capaz de validar fortificações próprias sem cutucada do Rick. No épico cockpit-ux-layout-filtros os 2 workers escalaram pr-aberto sem suite_status: digest ⚠ + pending-human + tab tmux vermelha juntos, exatamente como B3 prevê. No épico epico-idle-badge (pós-B1.2) o digest mostrou rotina silenciosa ✓ com suite_status: verde — primeira vez em 4 oportunidades que o ciclo passa limpo. O comparativo nos entries do digest do dia tornou a conclusão indiscutível.

v0.9.0 — Andaime endurecido pós-esteira corretiva

v0.9.0 2026-05-17 · commit dd77380 · sessão 2026-05-16 → 2026-05-17

Marco saindo da v0.8.0: ciclo de correções operacionais sobre falhas observadas durante o épico dark-mode e o teste real tz-cockpit-dashboard, mais um pós-tag de higiene (cleanup de docs/tasks, bucketing BRT) e a primeira validação em produção real das fortificações.

Saldo

10 PRs no mmb-andaime + 3 PRs no mmb-logger
3 épicos de produto entregues (dark-mode, tz-cockpit-dashboard, cockpit-ux-layout-filtros)
295 testes verdes no tooling (era 252; +43 novos nesta janela)
167 testes verdes no mmb-logger (+15 novos)
2 bugs reais do andaime detectados pelo próprio andaime sem cutucada do Rick (validação real de B2+B3)

O que mudou de fato

Contratos:

  • Schema semântico dos status no protocol.md: payload obrigatório por marco (issue-criada-N, pr-aberto-N, task-fechada-*), worker-master faz triagem determinística.
  • Convenção de fuso em source-of-truth.md: storage sempre UTC; agregação diária em /api/metricas/overview é BRT (UTC-3); não generaliza pra outros campos.

Mecanismos novos:

  • Hook inject-digest-tail.sh: marcos rotineiros do digest aparecem no próximo prompt do Mestre interativo (cursor monotônico, limite 10 por injeção).
  • Cron interno do reconcile no mmb-logger serve (a cada 300s) + endpoints GET /api/reconcile-status e POST /api/reconcile.
  • _enrich_epicos_closure no reconciler: épico fecha automaticamente ao ler Status: ✅ no master-briefing; reabre se ✅ é removido (estado derivado segue fonte canônica).

Endurecimento (B1/B2/B3):

  • B2: guard MMB_AGENT_ID != master nos hooks UserPromptSubmit — impede workers stateless de consumirem digest/pending-human destinados ao Mestre.
  • B3: atomicidade declarada da escala (digest ⚠ + write-pending-human.sh conjuntos) no profile do worker-master.
  • B1: reforço imperativo do schema pr-aberto-N no profile do orq local (tabela enumerativa + receita executável + caso real referenciado).

Higiene operacional:

  • review-cycle.sh não morre mais silencioso quando há errors sem resolves.
  • commd.sh respeita _MMB_TEST_MODE — teste de watchdog não polui mais journal real.
  • Janela interna do heartbeat_tick 60→120s, cobrindo latência pós-output do claude -p.
  • Cleanup automático de docs/tasks/*.md untracked em task-end/task-abort.
  • Bucketing BRT em /api/metricas/overview: storage UTC preservado, projeção do "dia operacional local" em runtime via modifier SQLite.

O que aprendemos

Reforço imperativo no profile não é contrato forte. B1 reincidiu nos 2 workers (cockpit + logger) mesmo após o reforço do PR #7 (warning ⚠ + tabela + receita executável + cláusula honesta + caso real). Critério registrado antes da reincidência (no body do próprio PR #7) acionou automaticamente a promoção pra wrapper obrigatório — o método se auto-disciplina quando o gatilho é explícito.
Hooks UserPromptSubmit rodam em qualquer claude da config — inclusive workers stateless. Foi anti-padrão estrutural descoberto em produção: cursor do digest avançava porque o claude -p do worker carrega os mesmos hooks. Guard por MMB_AGENT_ID != master resolve. Lição transferível: todo hook destinado ao Mestre interativo precisa provar contexto antes de produzir efeito colateral.
Atomicidade operacional precisa ser declarada, não presumida. Worker-master fazia a parte cheap (digest ⚠) e pulava a parte cara (write-pending-human.sh), deixando o Mestre cego. Wording explícito "ESCALA = digest ⚠ + pending-human" no profile eliminou o gap. Sempre que houver sequência de 2+ efeitos pra consequência semântica única, nomear como par atômico.
O andaime começou a auto-detectar bugs do próprio andaime. O épico cockpit-ux-layout-filtros (último desta janela) gerou 2 escalações pra pending-human sinalizando que B1 ainda não estava resolvido — sem cutucada do Rick, sem revisão humana adicional. B2+B3 validados em produção real: tab tmux ficou vermelha, digest mostrou ⚠, Mestre recebeu o sinal no turn seguinte.
