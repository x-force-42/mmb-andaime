# v0.1 — Robustez do canal de comunicação (Fase A: fortificar FS+tmux)

## Contexto

Smoke test do v0 (épico `cleanup-task-scripts`, 2026-05-14)
expôs 3 fragilidades estruturais do canal de comunicação:

1. **Ping perdido em "thinking":** `tmux send-keys ... C-m`
   chega no REPL do Claude quando ele está pensando; o texto
   bufferiza no input mas o Enter pode ser tratado como
   newline literal, não submit. Mensagem fica fria no inbox.
2. **Sem proteção contra writes concorrentes:** quando vários
   atômicos paralelos escreverem no mesmo inbox, `msg.sh` não
   serializa — torn writes possíveis se a linha exceder
   `PIPE_BUF` (4 KB no Linux).
3. **Atômicos zumbi:** Claude session pode crashar ou ficar
   travada; sem mecanismo, ninguém detecta. Worktree fica
   pendurada, registry de tasks fica mentindo.

Esta fase **não troca a stack** (FS + tmux + bash continuam).
Adota o modelo mental do actor model (mailbox-per-actor +
supervision tree + let-it-crash) na infra que já existe.
Prepara terreno pra v1 (Redis Streams) sem forçar migração.

## Intenção

Eliminar as 3 fragilidades acima e formalizar supervision
tree no nível de profile/script, mantendo retrocompatibilidade
com o método v0.

## Escopo

### Dentro

1. **Polling-on-every-turn** no protocol e profiles.
2. **`flock(1)`** em todos os writes do `msg.sh`.
3. **Agent registry** append-only em `.tooling/state/agents.jsonl`.
4. **Heartbeat** via touch de arquivo + supervision tick no
   profile do Orq de Projeto.
5. **Performativas FIPA estendidas** no protocolo
   (`propose`, `accept-proposal`, `reject-proposal`, `cancel`).
6. Atualização de `protocol.md`, `guardrails.md`, e 3 profiles
   pra refletir as mudanças.
7. Testes manuais de cada componente (não há suíte automatizada).

### Fora

- **`inotifywait` ativo.** Pode entrar em v0.2 se latência de
  polling for problema; v0.1 confia no polling do profile.
- **Migração pra Redis Streams.** Fica pra v1.
- **Atômicos longer-lived.** Permanece "atômico morre ao abrir
  PR" — escopo separado se virar útil.
- **Mudar `msg.sh` pra suportar agent-id hierárquico**
  (`core-X1`). Entra na v0.2 quando atômicos paralelos
  forem realmente spawnados; v0.1 só prepara o registry.
- **ACK obrigatório de mensagens.** Opcional via convenção
  futura, não exigido aqui.

## Componentes técnicos

### 1. Polling-on-every-turn

**Problema atacado:** ping perdido em thinking + sessão
recém-iniciada não vê mensagens "frias".

**Solução:** o protocolo passa a exigir que **toda sessão
Claude (master, orq, atômico) verifique seu inbox no início
de cada turn** — não esperar ping. Ping continua sendo
otimização de latência, não fonte da verdade.

**Implementação:**
- `protocol.md` ganha seção "Polling-on-every-turn é a
  garantia de entrega; ping é otimização".
- 3 profiles ganham 1 linha de bootstrap: "Antes de qualquer
  outra ação no turn, rode `ls -1t .tooling/inbox/<seu-id>/
  | grep -v '^\.'` e processe mensagens novas."
- `guardrails.md` ganha guardrail novo (M5/L8/A6 — numerar
  conforme camada): "Não pular polling do inbox no início
  do turn."

**Custo por turn:** 1 chamada `ls`. Desprezível.

**O que isso muda no smoke test de hoje:** caso similar
(ping cai durante thinking) seria recuperado no próximo
turn da sessão receptora, sem intervenção do Rick.

### 2. `flock(1)` em writes

**Problema atacado:** writes concorrentes ao mesmo inbox
podem corromper o arquivo de mensagem se ultrapassarem
PIPE_BUF.

**Solução:** `msg.sh` envolve a escrita em `flock` com
timeout curto, lock file por inbox de destino.

**Implementação:**
```bash
# trecho do msg.sh
INBOX_DIR=".tooling/inbox/$TO"
LOCK="$INBOX_DIR/.lock"
(
  flock --timeout 5 9 || { echo "lock timeout" >&2; exit 1; }
  printf '%s' "$CONTENT" > "$INBOX_DIR/$FILENAME"
) 9>>"$LOCK"
```

**Atomicidade reforçada:** writes < PIPE_BUF já são atomic;
flock cobre os > PIPE_BUF + garante ordem total entre
remetentes concorrentes.

**Custo:** ~ms por write. Lock file fica permanente como
`.lock` em cada inbox (ignorado pelo polling porque começa
com `.`).

### 3. Agent registry append-only

**Problema atacado:** não há fonte da verdade pra "quais
agentes existem agora". Sem isso, supervision não pode
existir.

**Solução:** registry append-only em
`.tooling/state/agents.jsonl`. Estado atual = redução do log
(igual journal da frente 1). Eventos:

```jsonl
{"ts":"2026-05-14T20:40:21Z","ev":"spawn","id":"core-X1","parent":"core","pane":"mmb:1.1","pid":12345,"task":"X1","epic":"cleanup-task-scripts"}
{"ts":"2026-05-14T20:50:09Z","ev":"deregister","id":"core-X1","reason":"pr-merged","pr":7}
{"ts":"2026-05-14T20:51:30Z","ev":"deregister","id":"core","reason":"session-ended"}
```

**Componentes:**
- `.tooling/state/agents.jsonl` (versionado? **não** — segue
  mesma lógica do inbox/intents: estado runtime, gitignore
  com `.gitkeep` preservando dir).
- `.tooling/bin/agents.sh` — comandos:
  - `agents.sh register <id> <parent> <pane> [task] [epic]`
  - `agents.sh deregister <id> <reason>`
  - `agents.sh list` — agrega o jsonl e mostra agentes vivos.
  - `agents.sh status <id>` — info de um agente específico.
- `spawn-atomic.sh` chama `agents.sh register` no final.
- `open-pr.sh` chama `agents.sh deregister` antes do
  `kill-pane`.
- `task-abort.sh` chama `agents.sh deregister` com reason
  `aborted`.

**Por que JSONL e não JSON mutável:** append-only é à prova
de race condition (com `flock`), audita-se trivialmente, e
é a mesma filosofia do journal (frente 1). Estado = reduce.

### 4. Heartbeat + supervision tick

**Problema atacado:** atômicos zumbi não detectados.

**Solução de heartbeat (atômico):**
- `.tooling/state/heartbeats/<agent-id>.alive` — arquivo
  vazio, mtime atualizado via `touch`.
- Atômico chama `agents.sh heartbeat` no início de cada turn
  (mesma chamada que faz polling).
- Combinação polling+heartbeat: 1 ação de inicialização do
  turn faz ambos.

**Solução de supervision (orq local):**
- Profile do orq de projeto ganha seção "Supervision tick".
- Quando o orq vê pings de filhos OU quando faz round de
  polling, ele opcionalmente roda `agents.sh check-children
  <orq-id> --threshold 600` (10 min sem heartbeat).
- Filhos travados → `task-abort.sh` automático + `error`
  pro mestre.

**Why threshold 10 min:** atômico em "baking" pode levar
minutos. 10 min é margem larga. Configurável em `config.sh`
(`MMB_HEARTBEAT_TIMEOUT=600`).

**Não automatizar com cron:** mantém na cadência de turns
do orq. Cron adiciona componente fora do tmux, complica
debug. Se cadência for insuficiente, vira candidato pra v0.2.

### 5. Performativas FIPA estendidas

**Problema atacado:** vocabulário atual (5 tipos) não cobre
casos comuns (negociação de escopo, cancelamento de task).

**Solução:** adicionar 4 performativas ao `protocol.md`:

| Type | Quem | Pra quem | Quando |
|---|---|---|---|
| `propose` | atômico ou orq | parent | "sugiro estender escopo de X" |
| `accept-proposal` | parent | filho | "aceito a proposta, prossiga" |
| `reject-proposal` | parent | filho | "rejeito; mantém escopo original" |
| `cancel` | parent | filho | "abandone a task atual" |

Não obrigatório. Vocabulário disponível. Existentes
(`briefing/question/answer/status/error`) continuam válidos.

**Documentar mapeamento informal → formal:**
- `briefing` ≈ FIPA `request`
- `question` ≈ FIPA `query-if` ou `query-ref`
- `answer` ≈ FIPA `inform-ref`
- `status` ≈ FIPA `inform`
- `error` ≈ FIPA `failure`

Pra observabilidade futura (frente 1 — diário de bordo),
performativa padronizada facilita queries cross-épico.

## Riscos e tradeoffs

| Risco | Mitigação |
|---|---|
| Polling adiciona ruído visual em pane idle | `ls -1t` curto, output curto; aceitável |
| flock pode falhar silenciosamente se FD esgotar | Timeout de 5s + erro explícito em stderr |
| Heartbeat via touch é mais frágil que heartbeat estruturado | Caso de v0.1; pode virar evento no JSONL em v0.2 se justificar |
| Performativas novas ficam só na doc, ninguém usa | Aceitável — vocabulário disponível, adoção orgânica |
| Threshold de 10 min pode ser curto pra atômicos com baking longo | Configurável; ajusta na primeira coleta de dados |
| Conversão de agentes existentes (master/core/cockpit/aquarium) pro registry | Bootstrap do `up.sh` registra todos os 4 orqs no startup |
