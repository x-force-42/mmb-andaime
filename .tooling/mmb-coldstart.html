Andaime MMB · cold start guide

up — primeiro voo em potência máxima

Como subir o andaime do zero, com Opus em todas as camadas, aquário renderizando, e uma feature concreta de teste pra exercitar o pipeline end-to-end pela primeira vez.

MMB_MODE=normal Opus 4.7 · effort high ~$0.50 estimado pro primeiro teste v0.5.0+

Visão geral

Esta página é o passo-a-passo do primeiro uso real do andaime — depois de toda a base de v0.3/v0.4/v0.5 estar paga. Não é tutorial inicial; assume que você já leu mmb-cli.html e mmb-testing.html.

O fluxo aqui pula intencionalmente todos os modos de teste — você não veio pra rodar smoke; veio pra orquestrar trabalho de verdade num dos três repos.

Pré-requisito mentalEsta é a primeira vez que você delega trabalho REAL pro andaime. Vai consumir tokens de Opus, mexer num repo de produto, abrir issue e PR no GitHub. Confirme que está em mood pra esse compromisso antes de subir.

O que é "potência alta"

A configuração de modelos por camada vive em .tooling/config.sh, controlada por MMB_MODE:

ModoMestreOrq localAtômicoPra quê
fastHaiku lowHaiku lowHaiku lowsmoke / iteração do andaime
balancedOpus highSonnet mediumSonnet hightrabalho real econômico
normalOpus highOpus highOpus highpotência máxima — este guia

normal é o default — se você não setar MMB_MODE, é o que vai rodar. Significa Opus 4.7 com effort high em master, workers (orq local) e atômicos. Heartbeat timeout em 600s, worker timeout em 1200s. É o mais caro e o mais capaz.

tradução práticaNão precisa exportar variável nenhuma. Só não rode MMB_MODE=fast que estará em potência máxima.

Cold start — 6 passos

1

Pré-checks

Confirma que o ambiente está pronto antes de subir nada.

$ cd /home/eliezer/llab/MMB
$ .tooling/bin/mmb preflight

Saída esperada: exit 0 — deps OK, gh autenticado, sem postmortems pendentes, sem branches task/* vivas, sem dirty state nos 3 repos.

Se exit 1 (warning), leia o motivo. Se for só "branches task/* vivas" de testes anteriores, considere rodar reset-all.sh --dry-run pra ver o que limparia — e se aprovar, reset-all.sh sem flags.

Se exit 2 (crítico), pare. Provavelmente gh não está autenticado ou alguma dep falta. Resolva antes de prosseguir.

2

Baseline verde no sandbox

Antes de tocar nada real, prova que o substrato está sadio:

$ .tooling/bin/smoke.sh hardening

~32s. Tem que terminar com === HARDENING PASS ===. Roda em sandbox isolado em /tmp/mmb-smoke-XXX/ — não toca seu inbox real.

Cobertura: L1-L6 (daemon lifecycle) + M1-M7 (falhas/recovery) + B1 (bridge regex) + AG1-AG9 (agents) + SP1-SP8 (spawn-atomic). Se algum cenário falhar, pare aqui e investigue — não tente subir o stack com sandbox vermelho.

3

Subir o stack (tmux + daemon)

Cria a sessão tmux mmb com todas as windows necessárias:

$ .tooling/bin/up.sh

O que isso faz:

  • Cria sessão tmux mmb; anexa se já existir.
  • Window 0 master: abre Claude Code interativo no diretório raiz. É onde você vai conversar com o Mestre.
  • Window 1 commd: roda o daemon de mensageria foreground.
  • Windows 2-4 core/cockpit/aquarium: tail -F dos logs dos workers — só observação.
  • Window 5 journal: tail do logs/journal.jsonl formatado com jq.
  • Window 6 aquario: roda a viz PixiJS do mmb-aquarium (server local).
  • Window 7 bridge: aquario-bridge.py publicando eventos no relay WS.

flag útilSe você já tem Claude Code aberto noutra janela e quer que ELE seja o Mestre, use .tooling/bin/up.sh --no-master-claude — a window 0 fica como zsh limpo, sem competir pelo papel.

4

Abrir o aquário no browser

A window aquario do tmux sobe um server local (provavelmente vite ou similar — verifique a saída). Pegue a URL que aparecer (algo como http://localhost:5173) e abra no browser.

Você deve ver a tela do aquário vazia. Quando o bridge começar a publicar eventos (próximos passos), criaturas vão aparecer.

teste rápido do canalAntes de seguir, confirme na window bridge que tem a linha ws: connected. Sem ela, o aquário ficará vazio mesmo com workers ativos.

5

Verificar saúde do stack

Numa shell fora do tmux (ou criando uma split):

$ .tooling/bin/mmb status

Tem que mostrar:

  • commd: RUNNING com pid
  • Inboxes todos com top=0
  • Sem branches task/* nos repos
  • Journal mostra último evento (provavelmente commd iniciado)

Se algo aqui estiver torto, pare. Se tudo verde, você está pronto pra delegar trabalho de verdade.

6

Abrir a intenção com o Mestre

Vá pra window master do tmux (Ctrl-b 0). Você está conversando com o Mestre Claude. Ele vai seguir o ciclo das 5 fases:

  1. Recepção — você descreve o que quer
  2. Discovery — Mestre lê o repo relevante read-only
  3. Briefing — Mestre te mostra o briefing pra aprovação
  4. Dispatch — após seu OK, Mestre dispara via msg.sh
  5. Acompanhamento — Mestre relata status conforme chegam

O importante: você só fala com o Mestre. Ele faz tudo via mailbox. Você acompanha pelo aquário (visual) e pelos logs (window journal, ou mmb journal --follow em outra shell).

Feature concreta pra testar

Adicionar README.md ao mmb-core

O mmb-cockpit tem README.md, o mmb-aquarium tem README.md, mas o mmb-core não tem nenhum. Isso é uma lacuna visível, isolada, e perfeita pra primeiro voo.

Por que essa feature é boa pro primeiro teste

  • Single-repo: só toca mmb-core. Não exercita fan-out (vem depois).
  • Escopo travado: 1 arquivo novo. Não tem mil maneiras de fazer.
  • Referências existentes: o atômico pode espelhar o estilo dos READMEs dos outros 2 repos.
  • Sem deps externas: nenhum pacote novo, nenhuma config nova, nenhum teste novo necessário.
  • Verificável em 30 segundos: você abre o PR, lê o README, mergeia ou pede ajuste.
  • Custo previsível: ~$0.30–$0.50 de Opus. Dentro de um cap razoável pra prova de conceito.

Critério de aceite

  • mmb-core/README.md existe na branch task/*
  • Cobre: o que é o projeto, principais arquivos do top-level, como rodar (referenciando pyproject.toml), link pro CLAUDE.md
  • Estilo coerente com READMEs dos outros 2 repos (tom, estrutura)
  • PR aberto no GitHub com label task e project:mmb-core
  • Sub-issue criada (e linkada no PR)

Como falar isso pro Mestre

Vá na window master do tmux e cole isso (ou diga em suas palavras — o Mestre vai pedir clarificação se precisar):

Quero delegar uma melhoria pequena no mmb-core: adicionar um README.md. Contexto: mmb-cockpit e mmb-aquarium têm README.md no top-level, mmb-core não tem. É uma lacuna visível. Escopo: - 1 arquivo novo: mmb-core/README.md - Cobrir: o que é o projeto (bot Discord, Garagem, Meeseeks, API REST), principais arquivos do top-level (bot.py, claude_runner.py, embeds.py, etc), como rodar (pyproject.toml), link pro CLAUDE.md. - Estilo coerente com os READMEs dos outros 2 repos. Critério de pronto: PR aberto no x-force-42/mmb-core, label task + project:mmb-core, sub-issue criada e linkada. Pode produzir o briefing e me mostrar antes de despachar.

o que o Mestre vai fazerLer os READMEs do cockpit e aquário pra ancorar estilo. Listar arquivos do top-level do mmb-core pra ter o que descrever. Produzir master-briefing.md em .tooling/intents/<data>-readme-core/. Te mostrar na conversa. Aguardar seu OK. Depois disparar via msg.sh core briefing readme-core ….

O que esperar visualmente

  1. ~5s depois do dispatch: aquário mostra um meeseek [W] worker-core-<pid> nascendo (born).
  2. ~10-30s depois: o worker do core (orq local) processa o briefing. Vai criar a issue no GitHub, spawnar um atômico via spawn-atomic.sh. O meeseek worker-core morre (died_happy).
  3. Aquário mostra um meeseek [A] atomic-… nascendo — esse é o atômico fazendo o trabalho de verdade. Ele vai ficar vivo enquanto heartbeating, com health ~1.0.
  4. 1-3 minutos: atômico edita arquivo, commita, push, abre PR via open-pr.sh. Meeseek atômico morre happy. PR aparece no GitHub.
  5. Imediatamente depois: outro meeseek worker-core nasce — o orq processando o status pr-aberto-N que volta pro Mestre.
  6. Mais 5-10s: meeseek worker-master nasce — o Mestre processando o status. Ele te conta na conversa que o PR está pronto.

Total estimado: ~3-5 minutos do dispatch ao PR pronto pra revisão.

Acompanhamento ao vivo

3 telas pra deixar abertas:

OndeO que olha
Browser (aquário)Criaturas nascendo, vivendo, morrendo. O fluxo visual completo.
tmux window masterConversa com o Mestre. Ele te conta o que está acontecendo conforme recebe status.
tmux window journalStream estruturado dos eventos: dispatch, claim, done, dead, worker-done. É a auditoria.

Se quiser ver detalhe de um worker específico, Ctrl-b 2/3/4 (core/cockpit/aquarium) ou em outra shell:

$ .tooling/bin/mmb logs core --follow

Revisão & merge

  1. Mestre te avisa quando o PR está aberto.
  2. Abra o PR no GitHub.
  3. Leia o README — verifica se cumpre o critério de aceite.
  4. Se OK: clique Squash and merge. Não use merge tradicional — o método foi desenhado pra squash.
  5. Se precisa de ajuste: comente no PR. O atômico já morreu — você pode pedir ao Mestre pra spawnar correção (ele faz outro briefing) ou editar manualmente.

após o mergeVolte ao tmux window master. Conte ao Mestre que mergiou. Ele provavelmente vai sugerir rodar task-end.sh pra limpar a worktree + branch local — pode aprovar.

Encerramento

Depois do PR mergeado e da worktree limpa:

$ .tooling/bin/mmb status
# confirma: inbox limpo, sem branches task/*, .dead vazio

Se quiser parar o stack inteiro (encerrar a sessão de trabalho):

$ tmux kill-session -t mmb
# mata tudo: commd, bridge, aquario server, master Claude, tails

Se quiser apenas o reset pesado (PRs abertos, branches, etc — útil só se algo deu errado):

$ .tooling/bin/reset-all.sh --dry-run   # revisa o que faria
$ .tooling/bin/reset-all.sh             # só se aprovou o dry-run

No final, você terá: 1 PR mergeado em mmb-core, 1 sub-issue fechada, journal com a história completa em .tooling/logs/journal.jsonl, e a sensação fundamental de que o andaime fez trabalho de verdade pela primeira vez.

Troubleshooting — quando algo trava

Mestre não responde / parece preso

Provavelmente está em "thinking" longo (Opus high effort). Aguarde até 30s. Se passar disso, Ctrl-c e tente reformular.

Nada aparece no aquário

Verifique nessa ordem:

  1. Window bridge mostra ws: connected?
  2. Window commd mostra RUNNING?
  3. Browser apontando pra URL certa (geralmente Vite na 5173 ou 8080)?
  4. mmb journal -n 20 mostra dispatches recentes?

Worker do core foi pra .dead

Significa que o orq local estourou timeout (default 1200s em normal mode) ou retornou exit não-zero. Diagnóstico:

$ mmb inbox dead
$ mmb logs core -n 100

Provavelmente o briefing exigiu coisa que o orq não conseguiu cumprir (e.g., autenticação, permissão). Conte ao Mestre o que aconteceu — ele vai propor refinamento do briefing.

Atômico ficou zombie (health caindo no aquário)

Heartbeat parou. Pode ter crashado silenciosamente. Em outro shell:

$ .tooling/bin/agents.sh list
$ .tooling/bin/agents.sh check-children core

Se confirmado morto, conte ao Mestre — ele decide se deregistrar manualmente e re-spawnar.

Custou mais do que esperado

Pare o stack (tmux kill-session -t mmb) e revise .tooling/logs/workers/<dest>.log pra ver quantos workers foram invocados e o que cada um fez. Pode ter tido loop não detectado de question↔answer.

expectativa de custoPrimeira execução em potência máxima: estimativa $0.30-$0.50. Se passar de $1, há algo errado — pare e investigue. Workers em loop ou briefings ambíguos são as causas mais comuns.
